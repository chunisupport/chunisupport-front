## 1. 最重要・基本ルール

### 開発環境とファイルエンコード
- **すべてのファイルは UTF-8 (BOMなし) エンコード**で記述・保存してください。
- 特に Windows (PowerShell) 環境でファイルを読み書きする際は、デフォルトのエンコードによる文字化けを防ぐため、必ず明示的に UTF-8 を指定してください。
  - 例: `Get-Content -Path "path/to/file" -Encoding utf8 -Raw`
- コメントや文字列が文字化けしていないか、作業後に必ず確認してください。

### 開発サーバー
- 開発サーバーは起動しないでください。すでにローカル環境で起動していることが多いためです。

### 実装・設計の基本原則
- **シンプルなコードの徹底**: 過剰な抽象化を避け、可読性が高くシンプルなコードを心がけてください。
- **車輪の再発明の禁止**: 一般的なユーティリティやアルゴリズムは自作せず、標準機能や信頼できる外部ライブラリを優先してください。
- **不要なチェックの排除**: 明確な理由がない限り、冗長な if 文による過剰なチェックは避けてください。

## 2. 共通化・ロジックの切り出し原則

フロントエンドにおける「重複コードの排除」と「一貫性の維持」のため、以下の切り出しルールを遵守してください。

### コンポーネントの共通化
- ボタン、入力フォーム、ダイアログなどのUI要素で、2回以上同様のデザイン・機能が登場する場合は、個別に実装せず `src/components/common` 等の共通コンポーネントとして定義、または Kobalte を拡張して一元管理してください。

### ロジック・状態の共通化 (Primitives / Utils)
- 複数のコンポーネントで共有される状態管理や、副作用を伴うロジックは、Solid.js のカスタム Primitive（`createCustomSignal` や `createAuth` などの関数）として `src/hooks` や `src/stores` に切り出してください。
- 純粋なデータ変形、計算ロジック、バリデーションなどは、コンポーネント内にインラインで書かず、`src/utils` に純粋関数（Pure Function）として切り出し、単体テストを容易にしてください。
- **注意**: 1回しか使われない、かつ極めて単純な処理を過剰に共通化（早期最適化）することは避けてください。

## 3. SolidJS アーキテクチャ＆ドメイン原則

あなたは SolidJS、Rsbuild、Webアプリケーション開発の専門家です。

### SolidJSに関する重要な注意
### SolidJSのリスト描画ルール
- **動的なリスト描画では `Array.prototype.map()` を使わず、`<For>` コンポーネントを使用**してください。
- 静的な一度きりの描画ではなく、状態変化で再評価される可能性がある配列はすべて `<For each={...}>` に統一してください。

- SolidJSはReactに似ていますが、**細粒度のリアクティビティモデル**を採用しており、**仮想DOMを使用しません**。
- コンポーネント関数自体は**最初の1回しか実行されない**ため、Reactの手癖（コンポーネント直下に生のアニメーションや副作用ロジックを置く、不用意なオブジェクトの分割代入でプロパティのリアクティビティを破壊するなど）で記述すると確実にバグになります。
- 知識だけに頼らず、必ず公式ドキュメントや提供されたリソースをこまめに検索・参照してください。MCP Server Context7 が利用可能である場合は、それを活用してください。

### マスターデータの制限
- マスターデータを直接使う箇所を減らしています。新規に直接参照するコードは原則禁止です。`versions` のように、必要最低限のものに個別に切り出された（カプセル化された）単位で利用してください。
- 既存コードに直接参照が残っている箇所は、保守時にカプセル化された単位への置き換えを検討してください。

### ドメインルール：難易度の文字列表記（大文字統一）
- 難易度を表すドメイン文字列（`BASIC`, `ADVANCED`, `EXPERT`, `MASTER`, `ULTIMA`）は、コード内、API通信、ローカルストレージ等のすべてにおいて**必ず大文字**で扱ってください。
- 難易度の比較・保存時は `toUpperCase()` を利用し、大文字で統一してください。
- クエリ文字列やURLパラメータなど、外部入力の正規化や外部出力の生成で小文字化が必要な場合に限り `toLowerCase()` の利用を許可します。ただし、正規化後にドメイン値へ変換する場合は必ず大文字へ戻して扱ってください。

## 4. コーディング規約 & UIライブラリの利用

### ライブラリの利用規約
- **UIコンポーネント**: **Kobalte** を利用してください。`button` などの基本的なHTMLタグに存在するものであっても、Kobalteのコンポーネントが存在する場合は、アクセシビリティ担保のために必ずKobalteのコンポーネントを優先利用してください。
- **アイコン**: **Lucide** を利用してください。利用できないと判断した場合は、必ず理由を説明して人間の判断を仰いでください。

### TSDocの付与
- 新規追加するコード、および改修を行った既存コードには、必ず **TSDocコメント**（目的、引数、返り値の説明）を追加してください（※既存の全ファイルに今すぐ追加する必要はありません。変更を施した箇所のみで必須です）。

### 定数・ハードコーディングの禁止
- 画面上の文言や設定値、マジックナンバーのハードコーディングは避け、専用の定数ファイル（例: `constants.ts`）に切り出して利用してください。
- 未使用の変数やインポート（Dead Code）は必ず削除してください。

## 5. 品質保証・テスト・レビュープロセス

### 変更時の必須ステップ
1. コードを書いたら、必ず `npm run check:ci` を実行して、スタイルガイドに準拠していることを確認してください。自分が変更していないファイルがエラーを出していても修正してください。
2. **ビルドエラーが発生したコードは絶対に受け入れられません。** 必ず本番ビルド（`npm run build`）が正常に通ることを確認してください。
3. 単体テストが存在する変更では、必ず `npm run test:unit` を実行してください。
4. CIやコマンドの出力を記した作業用の `txt` ファイル等は、確認後必ず削除してください。

### セルフレビュー・チェックリスト
作業完了後、提出前に自身のコードに対して以下のチェックリストを用いてセルフレビューを必ず実施してください。
- [ ] `npm run check:ci` と `npm run build` がエラーなく通っているか？
- [ ] （単体テストが存在する場合）`npm run test:unit` がエラーなく通っているか？
- [ ] デッドコード（未使用の変数、インポート、不要な引数）はすべて削除されているか？
- [ ] コンポーネントや関数に適切な TSDoc が付与されているか？
- [ ] Solid.js のリアクティビティ（Signal/Props）を不用意に破壊していないか？
- [ ] コメントや画面表示の文字列に文字化けはないか？
- [ ] APIの仕様や画面仕様に変更があった場合、関連ドキュメントに反映したか？

### テスト方針とテンプレート
- 重要なロジック（`src/utils` やカスタム Primitive）には、原則として **Given-When-Then** パターンを用いた単体テストを記述してください。
- 本リポジトリの標準実行コマンドは `npm run test:unit` です。
- 以下に、Node.js標準テストランナー（`node:test` / `node:assert/strict`）を使ったテストコードテンプレートを示します。

```typescript
import assert from 'node:assert/strict';
import test from 'node:test';
import { calculateDisplayScore } from './scoreCalculator';

// Test Case 1: 正常系
test('スコアが1,000,000以上の場合は理論値として計算されること', () => {
  // Given: 前提条件
  const rawScore = 1010000;
  const hasBonus = true;

  // When: 操作・実行
  const result = calculateDisplayScore(rawScore, hasBonus);

  // Then: 期待する結果 (アサーション)
  assert.equal(result, 1010000);
});

// Test Case 2: 異常系・境界値
test('スコアが負の値の場合はエラーをスローすること', () => {
  // Given
  const negativeScore = -500;

  // When & Then
  assert.throws(
    () => calculateDisplayScore(negativeScore, false),
    new Error('Invalid score value')
  );
});
```

## 6. Git運用ルール

### ブランチ名とコミットメッセージ
ブランチ名には `prefix/`（例: `feat/profile-page`）、コミットメッセージには `prefix: ` のプレフィックスを必ず付与し、本文は**日本語**で記述してください。

使用可能なプレフィックス：
- `feat:`（新機能追加）
- `fix:`（バグ修正）
- `docs:`（ドキュメントのみの変更）
- `style:`（コードの意味に影響を与えない変更、フォーマットなど）
- `refactor:`（リファクタリング、共通化など。機能変更なし）
- `test:`（テストの追加・修正）
- `chore:`（ビルドプロセスやツールの変更など）

⚠️ フロントエンドでは `update:`, `add:`, `remove:` などのプレフィックスは**使用禁止**です。

### ファイル移動
- ファイルを移動・改名する際は、コミット履歴を追跡可能にするため、必ず `git mv` コマンドを使用してください。
